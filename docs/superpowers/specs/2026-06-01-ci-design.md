# CI GitHub Actions — tests automatiques front + back

**Date :** 2026-06-01
**Statut :** validé (design), en attente du plan d'implémentation
**Périmètre :** ajout d'un workflow d'intégration continue. Aucun changement de code applicatif.

## Contexte

Le dépôt n'a aucune CI (`.github/workflows/` absent). Chaque `push` sur `main`
déclenche un déploiement Vercel (frontend) et Railway (backend) sans aucune
porte de tests automatique. Le code grossit (caisse, paie/CNPS, sync offline,
PWA) et embarque des tests critiques (concurrence, idempotence sync, paiement
mixte) qui ne sont aujourd'hui exécutés qu'à la main. Ce workflow exécute les
tests à chaque push/PR pour donner un signal ✓/✗ immédiat.

## Décisions de cadrage (validées)

1. **Périmètre = couverture complète**, y compris les tests d'intégration
   backend (service Postgres dans Actions + migrations + `.env.test`).
2. **Déclenchement = push `main` + pull_request, SANS blocage** (option 1).
   La CI est un voyant : elle ne bloque ni le push direct ni le déploiement
   Vercel. Le flux de push direct sur `main` est conservé. Durcissement
   ultérieur possible (protection de branche) sans toucher au workflow.
3. **Structure = un workflow, deux jobs parallèles** (`frontend`, `backend`)
   dans `.github/workflows/ci.yml` (approche A).
4. **Pas de lint bloquant, pas de `npm audit`, pas de build** dans la CI
   (build couvert par Vercel ; lint/audit produiraient une CI rouge dès le
   jour 1 sur des problèmes préexistants).

## Setup de test existant (référence)

- **Frontend** (`frontend/`) : `npm test` = vitest jsdom, 53 tests, sans DB.
  `npm run type-check` = `tsc --noEmit`. Lockfile : `frontend/package-lock.json`.
- **Backend** (`backend/`) : Prisma + PostgreSQL.
  - `npm test` = vitest, **exclut** `src/__tests__/integration/**` → suite
    « unit » (`api.test.ts`, `logic.test.ts`, `payroll.test.ts`).
  - `npm run test:integration` = `dotenv -e .env.test -- vitest run --config
    vitest.integration.config.ts` → suite intégration (`fileParallelism:false`,
    base de test partagée).
  - `npm run test:integration:setup` = `dotenv -e .env.test -- prisma migrate
    deploy`.
  - `.env.test.example` (committé) définit `DATABASE_URL`
    (`postgresql://restaurant:restaurant@localhost:5433/restaurant_test?schema=public`),
    `JWT_SECRET`, `JWT_REFRESH_SECRET` (valeurs de test factices),
    `SUPERADMIN_EMAIL`, `SUPERADMIN_PASSWORD`. Le vrai `.env.test` est local et
    git-ignoré.

## Architecture

Un fichier `.github/workflows/ci.yml`.

```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:
```

Deux jobs indépendants (`frontend`, `backend`) s'exécutant en parallèle, chacun
avec `actions/checkout@v4` + `actions/setup-node@v4` (cache npm). Node **20**
(socle LTS stable pour Actions ; ajustable).

### Job `frontend`

```yaml
  frontend:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: frontend
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: npm
          cache-dependency-path: frontend/package-lock.json
      - run: npm ci
      - run: npm run type-check
      - run: npm test
```

### Job `backend`

```yaml
  backend:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: backend
    services:
      postgres:
        image: postgres:18
        env:
          POSTGRES_USER: restaurant
          POSTGRES_PASSWORD: restaurant
          POSTGRES_DB: restaurant_test
        ports:
          - 5433:5432
        options: >-
          --health-cmd "pg_isready -U restaurant"
          --health-interval 10s --health-timeout 5s --health-retries 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: npm
          cache-dependency-path: backend/package-lock.json
      - run: npm ci
      - run: npx prisma generate
      - run: cp .env.test.example .env.test
      - run: npm run test:integration:setup
      - run: npm test
      - run: npm run test:integration
```

**Point clé — mapping de port `5433:5432` :** le service Postgres écoute en
interne sur 5432 mais est exposé sur l'hôte runner en **5433**, ce qui
correspond exactement au `DATABASE_URL` de `.env.test.example` (localhost:5433,
user/pass `restaurant`, base `restaurant_test`). Un simple `cp
.env.test.example .env.test` suffit — aucune réécriture d'URL, aucun secret.

Le service Postgres reste disponible pendant tout le job ; même un test « unit »
qui toucherait la base fonctionnerait.

## Secrets & robustesse

- **Aucun secret GitHub requis.** La base est éphémère (identifiants en clair
  acceptables) ; les secrets JWT de `.env.test.example` sont des valeurs de test
  factices, jamais celles de prod. Rien de sensible n'entre dans les logs.
- `--health-cmd pg_isready` garantit que Postgres est prêt avant les migrations
  (pas de course au démarrage).
- `npm ci` (pas `npm install`) → échec net si lockfile et `package.json`
  divergent.
- Chaque step échoue à la première commande rouge → badge ✗ + notification
  GitHub.

## Vérification

- **Avant commit** : reproduire localement le job backend (Postgres de test +
  `prisma migrate deploy` + `npm test` + `npm run test:integration`) pour
  garantir le succès en CI ; idem frontend (`type-check` + `npm test`).
- **Après push** : le workflow apparaît dans l'onglet Actions et s'exécute sur
  le push ; vérifier que les deux jobs passent au vert.
- La CI n'inclut que ce qui est déjà vert (53 tests front confirmés cette
  session ; suites back exécutées à la main).

## Hors périmètre (non fait)

- Lint bloquant, `npm audit` bloquant.
- Build / déploiement dans la CI (Vercel/Railway s'en chargent).
- Protection de branche / blocage du merge / blocage du déploiement Vercel
  (option 1 = pas de blocage).
- Matrice multi-versions de Node.
- Badge de statut dans le README.
